# 核心产品方向：高危 Nil 风险覆盖优先

本文档用于沉淀当前最重要、最贴近客户实际价值的产品方向。

结论很明确：

**LuaNilGuard 当前最该优先强化的，不是更远的“大模型语义增强”，而是对 Lua 最常见、最容易导致运行时崩溃的 Nil 高危模式做高质量覆盖。**

这条路线直接决定客户是否能在第一批真实试用中感知到价值。

## 一句话定位

LuaNilGuard 的核心价值，不是“抽象地理解 Lua 代码”，而是：

**优先发现那些最常见、最容易把线上业务打炸的 Nil 高危点。**

如果这条价值主线没有被做好，再复杂的长期语义架构，也很难在现实客户场景里建立信任。

## 为什么这是当前最好的产品方向

企业客户当前最关心的，不是系统理论上能否理解极复杂的动态语义，而是：

- 能不能抓到典型高危问题
- 能不能低成本接入
- 能不能在最常见的 Lua 代码习惯里立即看到价值

从这个角度看，当前最重要的风险面是：

1. `string.*` 库函数第一个参数可能为 `nil`
2. 字符串拼接 `..` 的任一侧可能为 `nil`
3. `pairs(x)` / `ipairs(x)` 的 `x` 可能为 `nil`
4. `#x` 的 `x` 可能为 `nil`
5. `nil` 数字参与顺序比较（`< <= > >=`）
6. `nil` 数字参与数值计算（`+ - * / % ^`）

这些问题有几个共同特点：

- 高频
- 高危
- 易于解释
- 用户一旦看到命中，就会立刻感知价值
- 对客户的“第一印象”影响极大

因此，从产品优先级看，它们应被视为：

**核心价值覆盖面（Core Nil Hazard Coverage）**

## 当前现状

### 已经覆盖较好的部分

当前默认 `sink_rules` 已支持：

- `string.match.arg1`
- `string.find.arg1`
- `string.gsub.arg1`
- `pairs.arg1`
- `ipairs.arg1`
- `length.operand`（`#`）
- `member_access.receiver`

这意味着：

- 字符串库的部分常见风险
- 表迭代
- 长度操作
- 成员访问

已经是正式能力的一部分。

### 仍然明显缺失的部分

目前最影响客户感知的缺口是：

1. 字符串拼接 `..`
2. 数值比较 `< <= > >=`
3. 数值计算 `+ - * / % ^`
4. 更多高频 `string.*` API 的扩面

这些缺口并不是“边角功能”，而是高危基础覆盖的直接缺口。

## 核心判断

### 判断一：先补高危基础覆盖，比先做更复杂智能更重要

在当前阶段，如果客户能明显看到：

- `string.find(x, ...)`
- `x .. y`
- `pairs(x)`
- `#x`
- `x < y`
- `x + y`

这些高危模式被稳定抓住，产品信任会快速建立。

反过来，如果这些典型模式缺失，再复杂的 LLM 能力也很难被认为“专业”。

### 判断二：这些高危模式应当默认启用，且解释清晰

这些模式不应被视为“高级可选规则”，而应作为：

- 默认规则
- 默认测试矩阵
- 默认文档演示场景

它们应成为用户理解 LuaNilGuard 的第一批代表性能力。

### 判断三：在这些高危模式上，应优先追求“偏强的可报告风险”

对于这些高危语言级操作：

- `..`
- `pairs/ipairs`
- `#`
- 比较
- 算术运算

如果静态层已经能确认操作数/参数存在明确的 nil 风险，应优先倾向：

- `risky`
- `risky_verified`

而不是过于保守地大量停在 `uncertain`。

因为这些场景：

- 语义清晰
- 风险直观
- 漏报的伤害往往高于适度保守升级

## 最推荐的近期产品路线

### Phase H1：Core Nil Hazard Coverage

这一阶段应专门面向“客户最常见的高危 Nil 风险”。

应包含：

1. 引入统一的 `binary_operand` 候选模型
2. 正式支持字符串拼接 `..`
3. 正式支持顺序比较 `< <= > >=`
4. 正式支持数值计算 `+ - * / % ^`
5. 扩展高频 `string.*` nil-sensitive API
6. 用客户真实风格（`module(..., package.seeall)` + 全局 `require(...)`）补完整测试矩阵

这一步的目标不是“更聪明”，而是：

**让工具对最典型、最重要、最能体现客户价值的高危模式形成默认强覆盖。**

## 最自然的实现方式

### 1. 通过 AST 新增统一的二元运算候选模型

当前 collector 主要覆盖：

- `function_arg`
- `receiver`
- `unary_operand`

下一步最合理的抽象是新增：

- `binary_operand`

用来统一承载：

- 字符串拼接
- 数值比较
- 数值运算

这比零散地为每个运算写一套特判要更干净、更可扩展。

### 2. 坚持 AST 路线，不回退到字符串匹配

既然正式分析路径已经强制依赖 Tree-sitter，那么这些高危二元运算也应：

- 在 AST 上做精确识别
- 不使用正则去扫 `..`、`+`、`<`

这样可以：

- 避免误扫注释/字符串
- 提高表达式边界准确性
- 为后续更细粒度的局部推理打基础

### 3. 用高危模板驱动默认测试与文档

这些模式不应只写成底层能力，而应被系统化地体现在：

- 默认测试
- MVP 样例
- README 示例
- 客户说明文档

因为它们本身就是产品价值表达的一部分。

## 推荐的高危模式清单

### 第一优先级（必须优先做好）

1. `string.find(x, ...)`
2. `string.match(x, ...)`
3. `x .. y`
4. `pairs(x)`
5. `ipairs(x)`
6. `#x`
7. `x < y`
8. `x <= y`
9. `x > y`
10. `x >= y`
11. `x + y`
12. `x - y`
13. `x * y`
14. `x / y`
15. `x % y`
16. `x ^ y`

### 第二优先级（应尽快补齐）

1. `string.sub(x, ...)`
2. `string.len(x)`
3. `string.byte(x, ...)`
4. `string.lower(x)`
5. `string.upper(x)`
6. `table.insert(tbl, ...)`

## 客户视角下的价值表达

如果这条路线做对，产品最有说服力的对外表述应是：

**LuaNilGuard 优先覆盖 Lua 中最常见、最容易引发运行时崩溃的 Nil 高危点。**

这比任何抽象的“智能代码理解”更容易让客户立刻理解价值。

## 当前阶段的产品原则

1. 优先补最常见的高危模式
2. 优先补客户一眼能理解的问题
3. 优先补会直接引发运行时错误的操作
4. 优先让“典型高危场景”可报告、可解释、可测试
5. 更复杂的智能增强必须服务于这条主线，而不是替代这条主线

## 最终总结

当前最重要的产品方向，不是“更复杂的智能”，而是：

**把 Lua 中最典型、最高危、最常见的 Nil 风险做成高质量的默认覆盖面。**

只有这层覆盖建立了，后续更复杂的语义增强和长期架构，才会真正被客户感知为“更强”，而不是“更复杂”。

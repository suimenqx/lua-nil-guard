# 长期最佳架构：从审查器到语义编译系统

本文档沉淀的是一个“不考虑短期实时交互体验，也不考虑 LLM token 成本”的长期最佳方向。

目标不是优化当前一次次的即时审查，而是让项目逐步演进为一个：

**面向整个仓库的语义编译系统（Semantic Compiler）**

这意味着：

- 不再把每个 case 当作一次性待审问题
- 而是先把整个仓库“编译”为可推理的语义中间层
- 再在语义层上进行风险推理、争议裁决和动态见证

## 核心判断

1. 长期质量的上限，不取决于一次单点判定的 prompt 有多强。
2. 长期质量的上限，取决于系统是否能持续沉淀、复用和验证“仓库语义”。
3. 因此，长期最优方向不是“让 LLM 更频繁参与”，而是“让 LLM 帮助系统把仓库编译成可推理的语义世界”。

## 为什么要从“审查器”转向“语义编译系统”

当前的基本形态仍然是：

- 找一个 sink
- 提取局部上下文
- 做静态分析
- 必要时扩容上下文
- 给出一次 verdict

这种模式可以在中小复杂度场景里工作良好，但长期会遇到天花板：

- 每个 case 都在重复理解同样的 helper
- 复杂语义没有被沉淀为仓库资产
- 不确定性无法系统性收敛
- 过程间理解会越来越依赖临时上下文

长期最优方案应当是：

- 先构建仓库级语义资产
- 再在其上做规则推理、风险查询和争议裁决

## 五层长期语义系统

### 1. 全仓语义索引层（Semantic Index）

这是最基础的一层，相当于编译器前端和事实仓库。

对整个仓库进行一次性结构提取，建立：

- AST
- CFG（控制流图）
- 简化 SSA / def-use
- 调用图（Call Graph）
- 模块 / `require` 关系图
- return-slot 图
- field-path 图
- alias 图
- 关键 side-effect 标记

这一步的职责不是直接给出风险 verdict，而是把“源码长什么样”转成“结构化、可查询的事实”。

长期价值：

- 为所有后续分析提供统一事实底座
- 避免每个 case 都重复从源码重新解析结构
- 为过程间推理、动态验证、语义缓存提供共享基石

### 2. 函数语义蒸馏层（Function Semantic Distillation）

这是 LLM 最适合大规模投入的地方。

目标是：对仓库中的函数建立“版本化、结构化”的语义摘要，而不是每次临时阅读源码。

每个函数都被蒸馏为可计算的语义对象，例如：

- 第 1 返回槽位是否可能为 `nil`
- 哪些分支触发 `nil`
- 哪些参数控制返回值是否为 `nil`
- 哪些参数经过该函数后可能变为 non-nil
- 是否是透明 wrapper
- 是否存在 defaulting 语义
- 是否只是包装另一个 helper

这层的输出不应该是自然语言长文，而应该是结构化 IR，例如：

```json
{
  "qualified_name": "user.lookup.resolve_name",
  "return_slots": {
    "1": {
      "may_be_nil": true,
      "nil_conditions": ["req.force_nil", "not req.cached_name and not req.profile"],
      "non_nil_conditions": ["req.cached_name", "req.profile"]
    }
  },
  "wrapper_kind": "none"
}
```

长期价值：

- 复杂 helper 不再需要每次临时重读
- 函数语义成为版本化资产，可缓存、可比较、可回放
- 为过程间推理提供比 AST 更高层的输入

### 3. 全仓固定点推理层（Whole-Repo Fixed-Point Inference）

当语义索引和函数语义摘要都存在后，系统就不该只停留在“一跳/二跳的手工 recognizer”，而应该做全仓固定点传播。

传播的对象包括：

- nil / non-nil 状态
- 条件性 non-nil
- return-slot 语义
- 参数到返回的依赖
- wrapper 语义
- field-path 风险
- guard 效果

也就是在整个调用图上反复传播这些状态，直到收敛。

这类似于：

- 数据流分析
- 过程间分析
- lattice / fixed-point 求解

长期价值：

- 减少对大量人工特判的依赖
- 让“多跳过程间语义”成为系统性能力，而不是零散补丁
- 真正消化复杂 helper、复杂传递链和条件性 nil 传播

### 4. 争议裁决层（Adversarial Semantic Review）

即使有了强静态和强语义层，动态语言里仍会有争议点。

这时 LLM 不应该直接“猜源码”，而应当：

- 基于已有语义图构造最强风险链
- 基于已有语义图构造最强安全链
- 再由第三方角色审查两条链是否都由语义事实支持

也就是说：

- LLM 不再自由阅读整仓源码并随意发挥
- 而是在“已经编译好的语义世界”里做证据审计

长期价值：

- 让 LLM 从“源码猜测器”升级为“证据裁判”
- 显著提升可解释性和稳定性
- 使复杂争议 case 的处理方式更接近审计流程

### 5. 选择性动态验证层（Selective Runtime Witness）

如果完全不考虑成本，长期最强方案一定包含“动态见证”。

对高价值、最高风险、最有争议的 case：

- 自动生成最小调用 harness
- 自动生成最小输入
- 在受控沙箱中执行目标函数
- 验证某个返回槽位是否可能为 `nil`
- 验证某条路径是否真的打到 sink

不是全仓运行，而是对“语义层仍有高价值争议”的 case 做精准见证。

长期价值：

- 将部分争议从“高可信静态推理”推进到“带实证的验证”
- 为最关键 case 提供更高可信度
- 强化系统在企业高风险场景下的说服力

## 最重要的洞见：把 `uncertain` 当作“语义债务”

短期工具往往把 `uncertain` 当作一种正常输出状态。

但长期最佳系统不应该这么看。

`uncertain` 应被视为：

**语义债务（Semantic Debt）**

每一个 `uncertain` 都意味着：

- 缺少某种函数语义
- 缺少某种 alias 解析
- 缺少某种 wrapper 模式
- 缺少某种 return-slot 依赖模型
- 缺少某种动态验证能力

因此，每个 `uncertain` 都应被归档到不同的治理队列：

- 新规则队列
- 新 contract 队列
- 新函数蒸馏队列
- 新动态验证队列
- 新语义索引增强队列

长期最佳系统不是“永远有很多 `uncertain`”，而是“`uncertain` 会持续被消灭并转化为仓库知识”。

## LLM 在长期架构中的最佳角色

长期最佳方案里，LLM 不应是单纯的“判案 agent”，而应是三种稳定角色。

### 第一次：函数蒸馏器（Function Distiller）

职责：

- 把复杂函数体蒸馏为结构化函数语义

产物：

- 函数级语义 IR
- return-slot 草案
- 参数依赖草案

### 第二次：争议审计器（Dispute Auditor）

职责：

- 在已有语义图上构造风险链与安全链
- 审核证据是否自洽

产物：

- 争议裁决
- 证据链完整性检查

### 第三次：规则蒸馏器（Rule Distiller）

职责：

- 从高频 unresolved / disputed case 中提炼新规则

产物：

- 新 AST pattern 草案
- 新 wrapper recognizer 草案
- 新 contract 草案
- 新动态验证建议

## 为什么“更强的 prompt”不是长期最优解

只靠 prompt 变强，解决不了以下核心问题：

- 输入不稳定
- 结论不可复用
- 历史知识不沉淀
- 不确定性不会自动收敛
- 审计性不足

而语义编译系统能从根上解决这些问题：

- 稳定性：依赖结构化事实，不依赖临时 prompt 拼接
- 复用性：函数语义和规则可以沉淀
- 可追溯性：证据链来自固定的语义图
- 自增长能力：系统可持续吸收新模式

## 最佳长期路线（简化版）

如果只保留一个长期方向，最优选择是：

1. 建全仓语义索引
2. 做版本化函数语义蒸馏
3. 做全仓固定点过程间推理
4. 用 LLM 做争议证据审计
5. 对少数高价值 case 做动态见证

换句话说：

**把 LuaNilGuard 从“审查器”，升级成“仓库级语义操作系统”。**

## 终极目标：仓库语义记忆

长期最有价值的产物，不应该只是每次输出一份报告，而应该是不断增长的“仓库语义记忆”：

- 哪些函数可能返回 `nil`
- 哪些 helper 是 guard
- 哪些 wrapper 会透传风险
- 哪些 field-path 经常出问题
- 哪些 suppressions 是合理的
- 哪些模式已经被实证验证
- 哪些模块积累了最多语义债务

一旦拥有这层记忆，系统就不再是“每次临时分析器”，而是：

**真正懂这个仓库的长期质量守卫者。**

## 一句话总结

如果彻底不考虑短期交互体验和 token 成本，长期最佳方案不是“让 LLM 更多参与每次判定”，而是：

**让 LLM 帮系统把整个仓库逐步编译成一个可推理、可记忆、可验证的语义世界。**

最终，越来越多的结论将不再依赖临时猜测，而是直接来自这个语义世界中的事实与推理。

# LLM 语义增强与精度闭环策略

本文档用于沉淀当前对“如何让 LuaNilGuard 在复杂场景下更智能，同时保持企业级可信度”的深度思考。

目标不是让 LLM 直接替代静态分析，而是让 LLM 在受约束、可复用、可审计的前提下提升复杂场景识别率。

## 核心判断

1. 现代 LLM 已经足够强，能够显著提升复杂函数、复杂分支、轻量过程间链路的语义识别率。
2. 但 LLM 不能单独承担企业级最终裁决，因为它不具备“稳定、可复现、可证明”的硬保证。
3. 因此，最佳路径不是“让 LLM 判案”，而是“让 LLM 做受约束的语义压缩、规则蒸馏与草案生成”，最终仍由静态引擎控制正式 verdict。

## 问题重述

在简单、本地、有界的场景中，当前静态引擎已经可以：

- 识别显式 guard
- 识别 field-path 风险
- 识别透明 wrapper / defaulting wrapper
- 识别有界跨文件 nil-return 风险

但在以下场景中，纯静态规则的边际收益会快速下降：

- 中间函数做了复杂条件重写
- 中间函数对返回值做了复杂变换而不是直接转发
- 存在局部别名、有限分发表、动态调用
- 需要多步过程间控制流理解

这些场景适合引入 LLM，但必须严格控制交互次数、作用范围和结果生效方式。

## 七条更优建议

### 1. Case-Centric Expansion，而不是 Function-Centric Expansion

不要采用“每遇到一个复杂函数就额外调用一次 LLM”的方式。

更好的做法是：

- 以“一个具体 case”为单位
- 只在这个 case 首轮仍为 `uncertain` 时
- 将当前 sink、局部上下文、静态证据、Top-K 相关 helper 一次性放进同一个 prompt
- 让 LLM 做一次“面向该 case 的语义扩容”

这样可以避免：

- 对单个 case 产生 3 到 5 次甚至更多次 LLM 往返
- 多次中间结论拼接带来的不稳定
- 交互延迟线性增长

推荐约束：

- 每个 case 最多只允许一次 LLM 语义扩容
- Top-K helper 建议为 1 到 3 个
- 超出预算时宁可保守 `uncertain`

意义：

- 保持用户体验
- 保持推理上下文完整
- 将“智能增强”成本控制在可接受范围内

### 2. 函数语义摘要要做成缓存资产，而不是每次现算

如果 LLM 被用于理解复杂函数，绝不能每次 review 都重复分析同一个 helper。

正确做法：

- 首次命中复杂 helper 时，才进行一次函数语义摘要
- 将摘要结果按函数体哈希缓存
- 后续同一函数直接复用，不重复调用 LLM

建议缓存键：

- 文件路径
- 函数限定名
- 函数体内容哈希

建议缓存内容：

- `return_slot_1_may_be_nil`
- `return_slot_1_depends_on_args`
- `nil_branch_conditions`
- `summary_confidence`

不要缓存冗长自然语言解释，应缓存结构化摘要。

意义：

- 大幅减少重复交互
- 降低 token 成本
- 提高同一仓库内结果稳定性

### 3. 优先做“批量语义收割”，不要把成本都放在在线交互

在线路径不适合承担大量 LLM 语义分析。

更好的做法是：

- 在后台治理路径（如 `proposal-analytics` 后）统计高频 unresolved helper
- 选出 Top-N 最常出现、最影响精度的 helper
- 批量做结构化函数语义摘要
- 将结果缓存为仓库资产或 proposal 草案

这样，在线 review 时：

- 多数复杂 helper 已经被预分析
- 用户不必在交互时等待额外 LLM 往返
- 工具会越来越像“懂这个仓库”的审查器

意义：

- 把用户等待成本转为后台治理成本
- 提高真实试用时的交互流畅度
- 更符合企业级使用体验

### 4. 让 LLM 生成“规则”，而不只是生成“单次结论”

对重复出现的复杂模式，不应该每次都让 LLM 重新读代码给结论。

更好的方向是：

- 让 LLM 从复杂 helper 中提炼出模式
- 产出：
  - 新的 AST pattern 候选
  - 新的 wrapper recognizer 候选
  - 新的 contract 草案
  - 新的 `maybe_nil_return` 规则草案

然后由系统做两步处理：

1. 作为 `ImprovementProposal` 保存
2. 人工或静态规则复核后，蒸馏回正式静态引擎

这意味着：

- LLM 的价值不只是“帮当前 case 判一次”
- 而是帮助系统“长出可复用的新能力”

意义：

- 系统越用越强
- 后续对相似模式的识别不再依赖重复调用 LLM
- 更符合长期工程化路线

### 5. 在线路径必须设硬预算：最多一次升级，不做无限追问

这是控制用户体验和稳定性的关键约束。

推荐策略：

- 默认不额外调用 LLM
- 仅在首轮 `uncertain` 且存在高价值相关 helper 时，允许一次升级
- 升级后仍无定论，则直接停止，保留 `uncertain`

不要：

- 为一个 case 再开第二轮函数级追问
- 让 helper 再继续触发子 helper 的多次递归追问
- 把在线路径变成无限扩容链条

意义：

- 防止交互次数失控
- 避免复杂 case 的延迟爆炸
- 保持系统行为可预测

### 6. 先做静态切片，再把最小相关上下文交给 LLM

LLM 在复杂代码里出错，很多时候不是因为模型不够强，而是因为上下文太杂。

因此，进入 LLM 前应先做“与当前目标最相关的最小切片”：

- 只提取和目标返回槽位相关的 `return`
- 只提取和目标参数相关的赋值
- 只提取影响 nil/non-nil 的条件分支
- 去掉无关日志、无关局部变量、无关副作用路径

也就是说：

- 不要把整个大函数直接丢给 LLM
- 要先把“函数理解任务”压缩成“最小语义片段”

意义：

- 更高的识别准确率
- 更低的 token 消耗
- 更少的模型误判

### 7. 引入“可信度分层”，不要让所有 LLM 结论直接生效

即使 LLM 产出了结构化语义摘要，也不能一刀切直接进入正式 verdict。

建议将 LLM 结果分成三层：

1. `硬可验证`
   - 可被静态规则复核
   - 可以进入正式风险/安全判断

2. `软高可信`
   - 高度可疑但静态层暂时无法完全验证
   - 进入 `proposal` 或 `needs_human`
   - 不直接升级为正式硬 verdict

3. `解释性线索`
   - 仅用于提示、排查、建议
   - 不参与正式状态判定

这样做的好处是：

- 充分利用 LLM 的理解能力
- 同时守住企业级可信边界
- 避免“一次聪明猜测”直接改变正式裁决

## 为什么不推荐“按函数逐个在线摘要”

这是最容易让系统“看起来更智能，实际更糟”的方案。

问题包括：

- 延迟随着 helper 数量线性增长
- 交互次数增加导致结果波动放大
- Token 成本迅速上升
- 用户体感恶化

因此，函数语义摘要不是不能做，而是应当：

- 尽量缓存
- 尽量批量后台化
- 在线最多做一次 case 级扩容

## 推荐的最佳组合路线

### 在线主路径

- 仍以当前静态引擎为主
- 先做本地静态分析、bounded recognizer、已有证据裁决

### 在线增强

- 仅当首轮 `uncertain`
- 且存在明确的 Top-K 相关 helper
- 触发一次 case-centric semantic expansion

### 后台成长

- 用 `proposal-analytics` 收集 unresolved/medium 热点
- 批量做函数语义收割
- 产出缓存与规则草案
- 将高频稳定模式蒸馏回静态引擎

### 长期目标

- 在线 LLM 次数越来越少
- 因为大部分价值已经被缓存、规则化、仓库记忆吸收

## 对当前项目最适合的下一阶段落地方向

如果要继续推进，最值得先做的不是“到处加 LLM 调用”，而是：

1. 设计一次 `case-centric semantic expansion` 的输入/输出 schema
2. 明确它只在什么条件下触发
3. 将其输出接入 `ImprovementProposal`
4. 允许只有“硬可验证”的结果进入正式静态能力

## 一句话总结

真正更智能的方向，不是“让 LLM 多想几次”，而是：

- 按 case 一次性扩容
- 让复杂函数语义变成缓存资产
- 让高频模式被蒸馏成静态规则
- 让 LLM 只花在少数高价值 unresolved case 上

这样才能同时做到：

- 更聪明
- 更稳定
- 更可复用
- 更像企业级产品

# plan2.md

# Core Nil Hazard Coverage 执行计划

本计划用于指导 LuaNilGuard 下一阶段的现实产品化推进。

核心目标不是继续发散功能，而是：

**将 Lua 中最常见、最容易造成运行时崩溃的 Nil 高危模式做成高质量、默认可用、可解释、可测试的正式能力。**

---

## 1. 总体目标

### 1.1 产品目标

让客户在首次真实试用中，明确感知到 LuaNilGuard 能稳定发现这些高危问题：

- `string.*` 字符串函数首参可能为 `nil`
- 字符串拼接 `..` 任一侧可能为 `nil`
- `pairs(x)` / `ipairs(x)` 的 `x` 可能为 `nil`
- `#x` 的 `x` 可能为 `nil`
- 数字顺序比较（`< <= > >=`）操作数可能为 `nil`
- 数值计算（`+ - * / % ^`）操作数可能为 `nil`

### 1.2 技术目标

构建统一的“高危表达式候选模型”，优先基于 AST 实现：

- 不依赖字符串匹配
- 不依赖 regex fallback
- 能与当前静态 proof / risk signal / adjudication 体系无缝衔接

### 1.3 非目标

本计划不优先处理：

- 更复杂的长期 LLM 语义编译
- 任意动态调用的完美识别
- 全仓性能重构
- 更复杂的自动 contract 发现

这些可以后续做，但不能抢占当前核心价值覆盖的优先级。

---

## 2. 设计原则

### 2.1 高危优先

优先补高频、高危、语言级会直接报错的模式，而不是边缘语义。

### 2.2 AST 优先

正式能力必须建立在 Tree-sitter AST 上。

### 2.3 统一抽象

不要为每个运算符做散乱特判，应引入统一的 `binary_operand` 候选模型。

### 2.4 风险偏强

对于语言级高危操作，只要已有足够明确的 nil 风险信号，应优先落到：

- `risky`
- 或 `risky_verified`

而不是大量停在 `uncertain`。

### 2.5 客户风格覆盖

新增测试必须覆盖：

- 普通局部写法
- `module("...", package.seeall)`
- 全局 `require("...")`
- 跨文件传递

---

## 3. 当前基线

### 3.1 已有覆盖

当前默认已支持：

- `string.match.arg1`
- `string.find.arg1`
- `string.gsub.arg1`
- `pairs.arg1`
- `ipairs.arg1`
- `length.operand`
- `member_access.receiver`

### 3.2 当前主要缺口

最重要缺口是：

1. 字符串拼接 `..`
2. 数字顺序比较 `< <= > >=`
3. 数值计算 `+ - * / % ^`
4. 高频 `string.*` API 扩面

---

## 4. 阶段划分

## Phase H1：统一二元运算候选模型

### 4.1 目标

新增统一候选类型：

- `binary_operand`

用于承载：

- 字符串拼接
- 数值比较
- 数值计算

### 4.2 实现项

1. 在 AST 解析层识别 Lua 二元表达式节点
2. 为二元表达式定义规范化模型
3. 在 `collector` 中新增对应 candidate 采集路径
4. 为二元表达式 candidate 设计稳定的 `case_id`

### 4.3 建议数据结构

候选至少应包含：

- 左操作数表达式
- 右操作数表达式
- 运算符
- line / column
- function_scope
- 当前关注的是哪一侧（便于复用现有 `CandidateCase` 风格）

### 4.4 验收标准

- 能在 AST 上稳定发现 `..`、比较、算术表达式
- 不误扫注释和字符串
- 现有功能不退化

### 4.5 测试要求

新增单测覆盖：

- 基础二元表达式提取
- 嵌套表达式中的目标操作数定位
- 注释/字符串不误识别

---

## Phase H2：字符串拼接风险（最高优先级）

### 4.6 目标

正式支持：

- `a .. b`

当左侧或右侧可能为 `nil` 时，判为高危。

### 4.7 实现项

1. 新增 `sink_rule` 配置项  
   建议：
   - `concat.left`
   - `concat.right`

2. collector 将 `..` 拆为左右两个可独立审查的高危位置

3. 静态分析：
   - 若某一侧已有明确 nil 风险信号，则产出强风险
   - 若某一侧已有明确 non-nil proof，则可抑制对应侧

4. adjudication / verification：
   - 应将这类风险视为语言级高危
   - 避免过度降到 `uncertain`

### 4.8 验收标准

- `x .. y` 能被识别
- 左右两侧都能独立给出结论
- 跨文件来源变量若有明确 nil 风险，也能传入此处

### 4.9 测试矩阵

必须覆盖：

1. 同文件：
   - `local s = nil; return s .. 'x'`
2. 右侧为 nil：
   - `return 'x' .. suffix`
3. 安全 defaulting：
   - `return (name or '') .. 'x'`
4. 跨文件来源：
   - helper 返回 string 或 nil，再参与拼接
5. 客户风格：
   - `module + require` 下的跨文件场景

---

## Phase H3：数字比较风险

### 4.10 目标

正式支持：

- `<`
- `<=`
- `>`
- `>=`

任一操作数可能为 `nil` 时，判为高危。

### 4.11 边界

不要把以下纳入该阶段高危比较：

- `==`
- `~=`

因为它们在 Lua 中对 `nil` 通常是合法比较，不应制造噪音。

### 4.12 实现项

1. 为四类顺序比较增加 `binary_operand` sink rule
2. 将左右操作数分别视作可能风险源
3. 与当前静态 risk signal / proof 体系打通

### 4.13 验收标准

- 四类顺序比较可被识别
- 不误伤 `==` / `~=`
- 缺少 guard 时，明确给出高危结论

### 4.14 测试矩阵

1. `if score > threshold then`
2. `if score <= limit then`
3. 左侧 nil
4. 右侧 nil
5. guard 后安全
6. 跨文件返回可能为 nil 的数字变量参与比较

---

## Phase H4：数值计算风险

### 4.15 目标

正式支持：

- `+`
- `-`
- `*`
- `/`
- `%`
- `^`

任一操作数可能为 `nil` 时，判为高危。

### 4.16 实现项

1. 为上述运算符增加 `binary_operand` sink rule
2. 与现有静态分析打通
3. 优先把“明确 nil 风险”的本地情况判到可报告级别

### 4.17 验收标准

- 常见数值运算都能识别
- 左右操作数都可独立呈现风险
- 不因字符串字面量或无关表达式误判

### 4.18 测试矩阵

1. `total = count + bonus`
2. `ratio = total / size`
3. `power = base ^ exp`
4. 左侧 nil
5. 右侧 nil
6. 跨文件来源可能为 nil 的数字参与计算

---

## Phase H5：扩展高频 string 库覆盖面

### 4.19 目标

在当前已有：

- `find`
- `match`
- `gsub`

基础上，继续补齐高频 nil-sensitive API：

- `string.sub.arg1`
- `string.len.arg1`
- `string.byte.arg1`
- `string.lower.arg1`
- `string.upper.arg1`

### 4.20 实现项

主要是：

- 增加默认 `sink_rules`
- 增加对应测试
- 在文档中纳入“默认覆盖的高危字符串函数”

### 4.21 验收标准

- 这些 API 开箱即用
- 与现有 `string.find/match/gsub` 行为一致
- 文档可直接向客户说明

---

## Phase H6：客户风格端到端测试矩阵

### 4.22 目标

确保“高危基础覆盖”不是只在理想示例里成立，而是在客户真实风格里稳定成立。

### 4.23 必须覆盖的风格

1. `module("...", package.seeall)`
2. 全局 `require("...")`
3. 跨文件 helper 传值
4. 层层调用传递
5. 本地变量中转

### 4.24 必须覆盖的模式

每种高危模式都至少有：

1. 同文件样例
2. 单文件 + 外部 helper 样例
3. 模块风格样例
4. 至少一个“应安全抑制”的反例

### 4.25 验收标准

- 所有新增高危模式都有端到端 fixture
- 与客户代码风格一致
- 默认评审路径可直接验证这些模式

## 5. 详细执行顺序

### Step 1

先实现 `binary_operand` 候选抽象，不要先零散加特判。

输出：

- 基础 AST 节点识别
- collector 接口
- 最小单测

### Step 2

先接入字符串拼接 `..`，作为第一批高危表达式能力。

原因：

- 业务最常见
- 风险非常直观
- 用户最容易立刻理解价值

### Step 3

接入顺序比较 `< <= > >=`

原因：

- 这是典型“数字 nil 高危”
- 规则边界清晰

### Step 4

接入数值计算 `+ - * / % ^`

### Step 5

扩一批高频 `string.*`

### Step 6

将上述所有能力补成客户风格端到端测试矩阵

### Step 7

更新用户文档：

- 默认覆盖哪些高危模式
- 推荐客户先用哪些示例验证

---

## 6. 风险与控制

### 6.1 主要风险

1. 二元表达式 AST 模型设计不稳，导致后续规则难扩展
2. 比较类规则误把 `==` / `~=` 纳入，造成噪音
3. 对复杂表达式过早做太多语义推断，导致误报
4. 新增高危模式但没有客户风格测试，导致“看起来支持，实际不稳”

### 6.2 控制原则

1. 先把 AST 抽象做稳
2. 只覆盖高危且语义明确的运算符
3. 保持 bounded reasoning，不做任意复杂表达式推断
4. 每加一个模式就补完整测试矩阵

---

## 7. 成功标准

### 7.1 技术成功标准

1. 新增 `binary_operand` 候选模型
2. `..`、顺序比较、算术运算全部可被 AST 正式识别
3. 全量测试保持通过
4. 客户风格测试矩阵补齐

### 7.2 产品成功标准

1. 用户可直接看懂工具在抓什么
2. 典型高危模式命中后，客户能立刻感受到价值
3. “为什么这个没抓到”的常见抱怨明显减少
4. 工具对 Lua Nil 高危点的覆盖面更像“生产工具”，而不是“研究型工具”

---

## 8. 最终目标陈述

完成本计划后，LuaNilGuard 应该可以对外明确宣称：

**它优先覆盖 Lua 中最常见、最容易引发运行时崩溃的 Nil 高危操作。**

这将成为项目面向客户的核心价值表达，也是后续更复杂语义增强的现实基础。
